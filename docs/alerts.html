Alerts & Webhooks

Threshold-based alerting, automatic evaluation, event acknowledgement, and outbound webhook delivery for agent observability.

Key concepts: Alert rules define conditions. When POST /alerts/evaluate runs, matching rules produce alert events. Events can be acknowledged, and webhooks deliver notifications to external systems (Slack, PagerDuty, custom endpoints).

Overview

The alerting subsystem is built around three components:

  1. Rules — declarative conditions like “fire when error_rate > 0.15 in the last 60 minutes.”
  2. Evaluation — an on-demand endpoint that checks all enabled rules against live session/event data and emits alert events for any breaches.
  3. Webhooks — outbound HTTP notifications dispatched when alerts fire (or on demand via test delivery).

AgentLens does not run a background scheduler by default. Call POST /alerts/evaluate from a cron job, a monitoring loop, or the dashboard to trigger evaluation.

Supported Metrics

Rules can target any of the following metrics, computed over a sliding time window:

MetricTypeDescription
total_tokensCounterSum of input + output tokens across all sessions in the window
avg_tokens_per_sessionGaugeMean token usage per session
error_rateRatio (0–1)Fraction of events that are errors
avg_duration_msGaugeAverage event latency in milliseconds
max_duration_msGaugeWorst-case event latency in the window
session_countCounterNumber of new sessions started
event_countCounterTotal events recorded
token_rateRateTokens per minute

Get the full list programmatically:

GET /alerts/metrics
# Returns: { metrics: ["total_tokens", "avg_tokens_per_session", ...] }

Alert Rules API

List rules

GET /alerts/rules?enabled=true&metric=error_rate

Optional filters: enabled (true/false), metric (one of the supported metrics).

Create a rule

POST /alerts/rules
Content-Type: application/json

{
  "name": "High error rate",
  "metric": "error_rate",
  "operator": ">",
  "threshold": 0.15,
  "window_minutes": 60,
  "agent_filter": "gpt-4-agent",
  "cooldown_minutes": 30
}
FieldRequiredDescription
nameYesHuman-readable label (max 128 chars)
metricYesOne of the supported metrics
operatorYesComparison: <, >, <=, >=, ==, !=
thresholdYesNumeric threshold value
window_minutesNoLookback window (default: 60, max: 10080 / 7 days)
agent_filterNoScope the rule to a specific agent name (max 256 chars)
cooldown_minutesNoMinimum gap between re-fires (default: 15, max: 10080)

Update a rule

PUT /alerts/rules/:ruleId
Content-Type: application/json

{ "threshold": 0.20, "enabled": false }

Partial updates are supported — send only the fields you want to change.

Delete a rule

DELETE /alerts/rules/:ruleId

Cascading: all alert events linked to the rule are also deleted.

Evaluation

POST /alerts/evaluate

Checks all enabled rules against current data. For each rule:

  1. Computes the metric value over the configured time window.
  2. If the threshold condition is met and the cooldown period has elapsed since the last fire, an alert_event is created.
  3. If webhooks are configured, they receive the alert payload.

Response example:

{
  "evaluated": 5,
  "triggered": 1,
  "alerts": [
    {
      "alert_id": "al_abc123",
      "rule_id": "rl_xyz789",
      "metric": "error_rate",
      "metric_value": 0.23,
      "threshold": 0.15,
      "triggered_at": "2026-04-28T12:00:00Z"
    }
  ]
}
Tip: Call /alerts/evaluate on a schedule (e.g., every 5 minutes via cron) for continuous monitoring. The cooldown mechanism prevents duplicate alerts.

Alert Events

List events

GET /alerts/events?acknowledged=false&rule_id=rl_xyz789&limit=50

Filters: acknowledged (true/false), rule_id, limit (default 50, max 200).

Acknowledge an event

PUT /alerts/events/:alertId/acknowledge

Marks the event as acknowledged with a timestamp. Idempotent — acknowledging twice is harmless.

Webhooks

Webhooks deliver alert payloads to external HTTP endpoints. They’re used for Slack notifications, PagerDuty integration, custom dashboards, and more.

List webhooks

GET /webhooks

Create a webhook

POST /webhooks
Content-Type: application/json

{
  "url": "https://hooks.slack.com/services/T00/B00/xxx",
  "events": ["alert.triggered"],
  "secret": "my-hmac-secret",
  "enabled": true
}

The secret field, if provided, is used to generate an HMAC-SHA256 signature in the X-AgentLens-Signature header for payload verification.

Update a webhook

PUT /webhooks/:webhookId
{ "enabled": false }

Delete a webhook

DELETE /webhooks/:webhookId

Test delivery

POST /webhooks/:webhookId/test

Sends a synthetic test payload to verify the endpoint is reachable. Rate-limited to prevent abuse.

Delivery history

GET /webhooks/:webhookId/deliveries?limit=20

Returns recent delivery attempts with status codes, response times, and any errors.

End-to-End Examples

1. Alert on high error rate with Slack notification

# Step 1: Create a webhook for Slack
curl -X POST http://localhost:3000/webhooks \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://hooks.slack.com/services/T00/B00/xxx",
    "events": ["alert.triggered"],
    "secret": "slack-verify-secret"
  }'

# Step 2: Create an alert rule
curl -X POST http://localhost:3000/alerts/rules \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Error spike",
    "metric": "error_rate",
    "operator": ">",
    "threshold": 0.10,
    "window_minutes": 30,
    "cooldown_minutes": 60
  }'

# Step 3: Evaluate (run on a cron schedule)
curl -X POST http://localhost:3000/alerts/evaluate

# Step 4: Check unacknowledged alerts
curl http://localhost:3000/alerts/events?acknowledged=false

2. Monitor token consumption per agent

curl -X POST http://localhost:3000/alerts/rules \
  -H "Content-Type: application/json" \
  -d '{
    "name": "GPT-4 token budget exceeded",
    "metric": "total_tokens",
    "operator": ">",
    "threshold": 500000,
    "window_minutes": 1440,
    "agent_filter": "gpt-4-agent",
    "cooldown_minutes": 1440
  }'

3. Detect latency degradation

curl -X POST http://localhost:3000/alerts/rules \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Slow responses",
    "metric": "max_duration_ms",
    "operator": ">",
    "threshold": 30000,
    "window_minutes": 15
  }'
