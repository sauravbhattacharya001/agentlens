Reliability & SLA

SLA targets, compliance tracking, agent scorecards, leaderboards, data retention, and postmortem analysis.

This page covers four interconnected systems: SLA targets define what “good” looks like; scorecards grade each agent; leaderboards rank agents competitively; retention policies manage data lifecycle. Together they form AgentLens’s reliability layer.

SLA Targets

SLA targets are per-agent, per-metric thresholds that define acceptable performance. AgentLens evaluates live session data against these targets to compute compliance scores.

Supported SLA metrics

MetricDescription
p50_latency_msMedian event latency
p95_latency_ms95th percentile latency
p99_latency_ms99th percentile latency
error_rate_pctError rate as a percentage (0–100)
avg_tokens_inAverage input tokens per session
avg_tokens_outAverage output tokens per session
max_duration_msMaximum allowed event duration
min_throughputMinimum events per evaluation window

Create / update a target

PUT /sla/targets
Content-Type: application/json

{
  "agent_name": "gpt-4-agent",
  "metric": "p95_latency_ms",
  "threshold": 5000,
  "comparison": "lte"
}

Comparisons: lte (≤), gte (≥), lt, gt, eq. Upserts on the (agent_name, metric) pair.

List targets

GET /sla/targets
GET /sla/targets?agent_name=gpt-4-agent

Delete a target

DELETE /sla/targets
Content-Type: application/json

{ "agent_name": "gpt-4-agent", "metric": "p95_latency_ms" }

Compliance Checks

POST /sla/check
Content-Type: application/json

{
  "agent_name": "gpt-4-agent",
  "window_hours": 24
}

Evaluates all SLA targets for the given agent over the specified window. Returns per-metric compliance with pass/fail status and the actual measured value.

{
  "agent_name": "gpt-4-agent",
  "window_hours": 24,
  "compliance_pct": 87.5,
  "metrics": [
    {
      "metric": "p95_latency_ms",
      "threshold": 5000,
      "comparison": "lte",
      "actual": 4200,
      "passed": true
    },
    {
      "metric": "error_rate_pct",
      "threshold": 5,
      "comparison": "lte",
      "actual": 7.3,
      "passed": false
    }
  ]
}

Each check is also persisted as a snapshot for historical tracking.

SLA History & Summary

Snapshot history

GET /sla/history?agent_name=gpt-4-agent&limit=50

Returns stored compliance snapshots ordered by creation date (newest first). Use for trend charts and audit trails.

Summary across agents

GET /sla/summary

Aggregates the latest compliance snapshot for every agent that has SLA targets configured. Useful for executive dashboards and fleet-wide health overviews.

Agent Scorecards

Scorecards provide a letter-grade assessment (A+ through F) for each agent based on multiple performance dimensions: completion rate, error rate, token efficiency, and latency.

List all scorecards

GET /scorecards?days=30

Returns a scorecard for every active agent in the lookback window.

Single agent scorecard

GET /scorecards/:agent?days=30

Grading scale

ScoreGradeIndicator
95–100A+● Excellent
90–94A● Great
85–89A−● Good
80–84B+● Above average
75–79B● Average
70–74B−● Below average
60–69C+/C/C−● Needs improvement
50–59D● Poor
<50F● Failing

Scorecards include: total sessions, completion rate, error count, average token usage, first/last seen dates, and per-dimension breakdowns.

Leaderboard

The leaderboard ranks agents competitively across five dimensions. Results are cached for 30 seconds to keep aggregation queries fast.

GET /leaderboard?sort=efficiency&days=30&limit=20&min_sessions=2

Sort dimensions

DimensionWhat it measuresDefault order
efficiencyTokens per successful completion (lower = better)Ascending
speedAverage event latencyAscending
reliabilityCompletion rate, error-free sessionsDescending
costEstimated dollar cost per sessionAscending
volumeTotal sessions processedDescending

Parameters: days (1–365, default 30), limit (1–100, default 20), min_sessions (minimum sessions to qualify, default 2), order (asc/desc, overrides default).

Postmortem Analysis

Automated postmortem reports for failed or degraded sessions. Analyzes event timelines, identifies error cascades, classifies severity, and computes cost impact.

Generate a postmortem

POST /postmortem
Content-Type: application/json

{ "session_id": "sess_abc123" }

Response includes:

  • Timeline — Key events (first, last, errors, slow calls) with elapsed-time markers
  • Severity — SEV-1 through SEV-4 based on error rate thresholds
  • Root cause candidates — The first error event with context
  • Cost analysis — Token consumption and estimated dollar cost for the session
  • Tool call breakdown — Which tools were invoked and which failed

List recent error sessions

GET /postmortem?min_errors=3&limit=20

Returns sessions that have at least min_errors error events, suitable for triage and batch postmortem generation.

Severity classification

LevelError rate thresholdMeaning
SEV-1≥ 50%Critical — majority of events failed
SEV-2≥ 25%High — significant error rate
SEV-3≥ 10%Medium — elevated errors
SEV-4< 10%Low — isolated failures

Data Retention

Configurable data lifecycle management. Purge old sessions and events to control database growth while protecting tagged sessions from deletion.

View current config

GET /retention/config

{
  "max_age_days": 90,
  "max_sessions": 0,
  "exempt_tags": [],
  "auto_purge": false
}

Update retention policy

PUT /retention/config
Content-Type: application/json

{
  "max_age_days": 60,
  "exempt_tags": ["production", "incident"]
}

Sessions tagged with any exempt_tags value are never purged, regardless of age.

Database statistics

GET /retention/stats

Returns: total sessions, total events, oldest session date, database size estimate, and counts of sessions eligible for purge under current settings.

Manual purge

POST /retention/purge
Content-Type: application/json

{ "dry_run": true }

With dry_run: true, returns what would be deleted without actually removing anything. Set to false (or omit) to execute the purge.

Caution: Purge operations are irreversible. Always run a dry-run first and verify the counts before executing.

Recommended Workflow

  1. Define SLA targets for each production agent (PUT /sla/targets).
  2. Set up alert rules for the most critical metrics: error rate, latency, token budget (POST /alerts/rules).
  3. Configure webhooks for Slack/PagerDuty notifications (POST /webhooks).
  4. Schedule evaluation — call POST /alerts/evaluate and POST /sla/check every 5–15 minutes via cron.
  5. Review scorecards weekly to track agent health trends (GET /scorecards).
  6. Investigate incidents using postmortem analysis (POST /postmortem).
  7. Monitor leaderboard to compare agent efficiency and identify regression (GET /leaderboard).
  8. Configure retention to keep database size manageable (PUT /retention/config).
